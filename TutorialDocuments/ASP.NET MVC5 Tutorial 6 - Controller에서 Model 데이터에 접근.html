정의된 Key가 존재하지 않음



우리가 진행한 Tutorial 코드로 위의 에러는 반드시 발생 합니다. 원인은 아래의 코드에 있습니다.
GameServer/Models/UserInfo.cs

using System;
using System.Data.Entity;

namespace GameServer.Models
{
    public class UserInfo
    {
        public int UserID { get; set; }
        public string UserName { get; set; }
        public DateTime LastestConnectDate { get; set; }
    }


    public class UserInfoDBContext : DbContext
    {
        public DbSet<UserInfo> Users { get; set; }
    }
}

public int UserID { get; set; } 코드가 문제 입니다.
Model로 활용되는 Class에 변수명이 ID가 들어가게 되면 오류를 일으킵니다. 
자세한 내용은 https://msdn.microsoft.com/en-us/library/system.data.entity.modelconfiguration.conventions.idkeydiscoveryconvention(v=vs.103).aspx 혹은 https://gilesey.wordpress.com/tag/entitytype-has-no-key-defined/ 를 확인하시기 바랍니다.

해결 방법은 두가지가 있습니다.
  1. UserID 변수명을 수정
  2. Key Attribute 추가

우리의 Tutorial에서는 2번의 방법을 사용 했습니다.

수정된 코드는 아래와 같습니다.
GameServer/Models/UserInfo.cs

using System;
using System.ComponentModel.DataAnnotations;
using System.Data.Entity;

namespace GameServer.Models
{
    public class UserInfo
    {
        [Key]
        public int UserID { get; set; }
        public string UserName { get; set; }
        public DateTime LastestConnectDate { get; set; }
    }

    public class UserInfoDBContext : DbContext
    {
        public DbSet< UserInfo> Users { get; set; }
    }
}
[Key] Attribute 사용법이나 내부 구현이 궁금하신 분들은 C# Attribute 라는 검색어로 구글링하시면 쉽게 이해하실 수 있습니다.

강력한 Model Type과 @model 키워드
일찍이 우리 튜토리얼[ASP.NET MVC Tutorial 3 - View 추가]에서 ViewBag Object를 통해 Controller가 View Template에 Data를 전달하는 것을 살펴본 적이 있습니다. 
ViewBag Object는 Dynamic Object로 View에 정보를 전달하는 편리한 방법을 제공 해 줍니다. [late-bound는 C++에서의 Dynamic Binding 정도로 정의할 수 있겠습니다 - Runtime에 객체의 역할이 결정됨]

또한 MVC 는 strongly typed object를 View Template에 전달 할 수도 있습니다. 이 방법은 Code의 보다 강력한 Compile Time Check를 가능하게 하고, Visual Studio에서 지원하는 IntelliSense를 활용할 수도 있습니다.
이번 Tutorial의 첫 부분에서 Controller를 생성할 때 Add Scaffold Diaglog가 떴던 것을 떠올려 보세요. 바로 Scaffolding Mechanism이 바로 이 형식을 활용해 UserInfoController Class와 View Template을 생성 한겁니다.

그럼 이번에는 Controllers\UserInfoController.cs 파일에 생성된 Details 함수를 살펴보도록 하겠습니다.

public ActionResult Details(int ? id)
{
    if (id == null)
    {
        return new HttpStatusCodeResult(HttpStatusCode .BadRequest);
    }
    UserInfo userInfo = db.Users.Find(id);
    if (userInfo == null)
    {
        return HttpNotFound();
    }
    return View(userInfo);
}
매개변수인 id 값은 일반적으로 Route Data로서 전달 됩니다. 예를들어 http://localhost:xxxx/userinfo/details/1 URL에 대해 Controller에는 UserInfoController, Action Method에는 details, id에는 1이 매칭되도록 합니다. 또한 아래와 같은 방식으로도 전달 가능 합니다.
http://localhost:xxxx/userinfo/details?id=1

만약 userinfo를 찾았다면, 해당하는 userInfo Model의 Instance(Model에 실제 값이 들어간 객체)가 Details view로 전달 됩니다.

return View(userInfo);
계속해서 Views\UserInfo\Details.cshtml 파일을 살펴 보도록 하겠습니다.

View Template 파일 상단에 @model 구문을 추가하면 View에서 다룰 Object의 Type을 지정할 수 있습니다. Visual studio가 UserInfo Controller를 생성할 때 자동으로 Details.cshtml 파일에 @model 구문을 추가 했습니다.

@model GameServer.Models. UserInfo
이 @model 구문의 추가로 Controller에서 View로 전달된 UserInfo 정보에 접근할 수 있게 됩니다. 예를들어 Details.cshtml 파일의 View Template 코드들도 Model 객체를 이용해 UserInfo 정보를 DisplayNameFor 와 DisplayFor 함수에 전달 합니다. 또한 Create, Edit 함수와 View Template들도 UserInfo Model Object를 전달합니다.

Index.cshtml  View Template 파일과 UserInfoController.cs 의 Index Method를 살펴보겠습니다. Index Action 함수에서 View Template을 호출할 때 매개변수를 잘 보시기 바랍니다. 생성된 User 리스트를  Index Action 함수에서 View Teamplate으로 전달하는 코드입니다.

public ActionResult Index()
{
     return View(db.Users.ToList());
}

Controller를 생성할 때 Visual Studio는 자동으로 Index.cshtml 파일의 최상단에 아래의 @model 구문을 추가합니다.

@model IEnumerable<GameServer.Models. UserInfo>
이 @model 구문 역시 Model Object를 사용하여 Controller에서 View로 전달된 UserInfo 리스트에 접근할 수 있게 해줍니다. Index.cshtml ViewTemplate은 foreach 문을 사용하여 Model Object List에 대한 루프를 수행하게 합니다.
@foreach (var item in Model) {
    <tr >
        <td>
            @Html.DisplayFor(modelItem => item.UserName)
        </td>
        <td>
            @Html.DisplayFor(modelItem => item.LastestConnectDate)
        </td>
        <td>
            @Html.ActionLink("Edit", "Edit", new { id=item.UserID }) |
            @Html.ActionLink("Details", "Details", new { id=item.UserID }) |
            @Html.ActionLink("Delete", "Delete", new { id=item.UserID })
        </td>
    </tr >
}
위의 Code에서 Model Object는 IEnumerable<GameServer.Models.UserInfo>이므로 UserInfo Type으로 바로 Casting 된다고 볼 수 있습니다. (역주 : C#의 모든 자료형(Struct,Class,DefaultDataType)이 Object로의 치환이 가능합니다.) Index.cshtml의 Code는 Compile 타임에 에러체크가 가능하며 Intelisense의 지원도 받을 수 있습니다.


SQL Server LocalDB를 이용해 작업하기


첫 시작 시 아무런 Database도 구축되어 있지 않으므로 Entity Framework Code First가 자동으로 UserInfo  Database를 생성 합니다. Solution Explorer의 App_Data 폴더를 확인 해 보면 UserInfo.mdf가 생성되어 있는걸 확인할 수 있습니다. 아무것도 없다면 빨간 네모박스 된 아이콘을 클릭하세요. 모든 파일을 보여준다는 아이콘입니다. 


UserInfo.mdf 파일을 더블클릭해 SERVER EXPLORER를 여세요. 그리고 Table폴더의 UserInfo Table을 우 클릭하여 Show Table Data를 클릭 하세요. 여기에서 확인할 수 있는 한가지가 있습니다. Models\UserInfo 클래스에 앞서 설명드린대로 [Key] Attribute 를 사용해서 UserID 항목을 추가 했습니다. 해당 UserID값이 Table에서 Data를 관리할 Key값으로 자동 등록 되어 있습니다. Entity Framework 가 ID란 이름이 포함된 변수에 대해 자동적으로 키데이터로 변환시키기 때문이며 UserInfo를 추가할 때 UserName과 LastestConnectData만 입력하는 이유이기도 합니다. Entity Framework와 MVC에 대한 좀더 자세한 설명은 MVC and EF를 참조하시면 됩니다.



UserInfo Table을 다시 우 클릭하여 Open Table Definition을 누르세요. Entity Framework가 자동으로 생성한 Table 구조를 확인할 수 있습니다.


UserInfo Table과 UserInfo class가 서로 어떻게 매핑되는지 주의깊게 살펴보세요. Entity Framework Code First는 우리가 생성한 UserInfo 클래스를 기반으로 이 스키마를 자동 생성합니다. 

작업을 끝냈으면 UserInfoDBContext  우 클릭하여 Close Connection을 누르세요. 안닫으면 다음번 작업시 오류가 날 수도 있습니다. ( 컴퓨터에서 USB를 빼기 전 윈도우에서 이 장치를 제거 버튼을 클릭하는것과 비슷한 느낌)
